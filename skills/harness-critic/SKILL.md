# harness-critic：过程评审器

## 定位

经验沉淀三步法中的第二步（周期提炼）。输入是 `harness/trace/` 的原始失败记录，输出是结构化的模式报告和硬度升级建议。

三个核心问题：
1. 最近失败集中在哪些类型？
2. 有没有重复出现的模式？
3. 哪些经验应该升级为硬约束？

## 前置条件

项目必须已初始化 harness 目录结构（`harness/trace/` 存在且非空）。

如果 `harness/trace/` 为空或不存在，提示用户先使用 harness-init-java 初始化项目，或在日常开发中积累 trace 记录。

## 工作流程

### Step 1: 收集失败记录

扫描 `harness/trace/` 下所有 `.md` 文件，提取每条记录的五个要素：

- **日期** — 什么时候发生的
- **问题类型** — 编译失败 / 测试不通过 / 分层违规 / 配置错误 / 其他
- **根因** — 为什么发生
- **修复方式** — 怎么修的
- **状态** — 待处理 / 已写入文档 / 已纳入硬约束

分类标准：

| 类别 | 判断依据 |
|------|---------|
| 分层违规 | import 跨层、Controller 调 DAO 等 |
| 编译失败 | 类型不匹配、依赖缺失等 |
| 测试不通过 | 用例断言失败、Mock 不对等 |
| 配置错误 | 碰了受保护文件、硬编码等 |
| 业务逻辑 | 违反业务规则、状态流转错误等 |

### Step 2: 频率分析

统计每类问题的出现次数，识别 Top 3 高频问题。

对每类高频问题，检查是否跨多个 change 反复出现（跨 change 的模式比单 change 内的更有升级价值）。

### Step 3: 硬度升级建议

对每条记录，判断是否满足升级条件：

| 当前状态 | 升级条件 | 升级目标 |
|---------|---------|---------|
| 待处理 | 同类问题出现 2 次以上 | 写入 CLAUDE.md 或 docs/ |
| 已写入文档 | 同类问题出现 3 次以上 | 升级为 lint 规则或 Hook |

升级判断的颗粒度：同类问题 = 相同问题类型 + 相同根因模式。比如"跨层调用"出现 3 次，但如果根因不同（一次是不知道规则、一次是 Agent 幻觉、一次是故意绕过），不应该合并为同一模式。

### Step 4: 输出报告

按 `references/report-template.md` 模板生成报告，包含：

1. **概览** — 分析周期、记录总数、涉及 change 数
2. **Top 问题** — 高频问题列表（含次数和占比）
3. **模式识别** — 跨 change 反复出现的模式
4. **硬度升级建议** — 建议升级的条目及理由
5. **行动项** — 具体要做什么、谁来做、优先级

报告输出到终端，同时追加到 `harness/memory/lessons-learned.md`。

### Step 5: 确认和执行

将行动项逐条展示给用户，由用户决定：
- **立即执行** — 自动更新 CLAUDE.md / docs / lint 规则
- **标记待办** — 记录但不立即执行
- **忽略** — 不升级，记录忽略原因

## 注意事项

1. **只分析，不自动修改**。Step 5 的所有修改必须用户确认。
2. **不读取配置文件内容**（与 harness-init 保持一致）。
3. **分析范围可控**：默认分析全部 trace，用户可指定日期范围或特定 change。
4. **空 trace 时给出建议**：提示用户如何建立 trace 记录机制。

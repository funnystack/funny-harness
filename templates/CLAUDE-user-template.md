## 5. 我的编码偏好

### 5.1 数据库设计偏好

- 命名: 表名/字段名使用 `snake_case` (小写下划线)。
- 规范: 主键必须为bigint(20) 自增类型, 每个表必须要有`created_stime`(默认为当前时间), `modified_stime`(自动更新为当前时间), `is_del` (逻辑删除)。
- 手机号、密码、用户真实姓名、身份证号、银行卡号、个人住址、员工姓名、公司税务标识、属于 隐私信息，必须设计为加密存储 字段后缀为 \_encrypt，例如 mobile_encrypt, 对于手机号需要特殊增加 \_hash 字段 , 比如mobile_hash,phone_hash

### 5.2 业务开发代码偏好

- 涉及下单、支付、退款、结算、活动、抵扣券、优惠券、积分、支付、会员等功能时,需要考虑服务的幂等性,加分布式锁解决,Java类为`RedisLock`
- 多表操作强制开启事务，但务内严禁包含RPC 调用、MQ 发送、Redis 操作，防止事务挂起时间过长。
- 调用远程接口的时候需要打印调用前url、参数、header(如果有的话),调用后需要打印出接口的返回结果。
- 如果项目使用了多个数据源，禁止出现SQL语句中多库的联表查询,
- service上如果有`@Transactional(rollbackFor=*.class)` 的时候需要检查调用的mapper是否都有`@DS`注解,可能会导致数据源无法切换。
- Mapper返回使用List、Map等类型 ,必须创建对应的实体DTO.
- 如果遇到一种[业务逻辑]的调用方大于2个，且业务逻辑里的判断条件大于3个，则必须抽象成单独的service方法。
- 在更新业务数据时，禁止直接把查出来的完整实体传入更新。必须新建一个实体对象，只设置 `id` + 需要更新的字段，再调用修改方法。

### 5.3 代码风格偏好

- API Style: 禁用标准 RESTful（避开复杂的 PUT/PATCH），统一使用 GET/POST。所有响应封装在 `Result<T>` 中，由 GlobalExceptionHandler 统一拦截。
- 架构风格: 严格执行 Controller -> Service -> DAO 的分层架构。Controller层只做参数校验和编排，Service层负责业务逻辑处理，DAO层负责中间件(数据库、缓存等)的操作。
- Service层禁止使用mybatis-plus的QueryWarpper，尽量使用mybatis-plus自带的Mapper的方法支持，复杂查询必须使用Mapper自写SQL的方式实现。
